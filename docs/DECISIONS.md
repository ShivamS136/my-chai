# DECISIONS.md — Architecture Decision Records

Format: context → decision → consequences. To change a decision, add a new ADR superseding it; don't edit history.

---

## ADR-001: Static-only, no backend, forever
**Context:** Payment confirmation, donor lists, and recurring support all require a server + PSP integration, which reintroduces commissions, KYC, uptime liability, and the trust problems of the central platforms we're replacing.
**Decision:** Pure static output. No API routes, serverless functions, or databases — including "just a tiny function for X". Features requiring a backend are rejected at scope, not implemented carefully.
**Consequences:** No payment confirmation ever (marketed as the feature it is: "can't take a cut of what we never touch"). Free hosting everywhere. The template stays forkable by non-experts.

## ADR-002: UPI P2P intent URIs only — no PSP, no merchant flows
**Context:** PSPs (Razorpay etc.) give confirmation and reliable deeplinks but charge ~2% and require KYC/settlement accounts. NPCI merchant QRs (`mc` param) trigger verification checks against unregistered VPAs.
**Decision:** Emit plain `upi://pay` P2P URIs with only `pa, pn, am, cu, tn`. Never add `mc`, `tr`, `mode`, `purpose`, or signature params.
**Consequences:** Deeplink flakiness on GPay/PhonePe is accepted and mitigated by UX (ADR-006). Zero fees preserved. Equivalent legal posture to a printed QR at a chai stall.

## ADR-003: Config as a typed TS file, validated by Zod, `.strict()`
**Context:** Options were env vars, JSON, YAML, or TS. Content is nested (works, socials); creators are devs-first; typos must fail loudly.
**Decision:** `chai.config.ts` with `defineConfig()`; Zod strict schema; build fails with per-field actionable errors. Env vars only for analytics keys.
**Consequences:** Autocomplete + comments for creators; schema is the public API and versioned as such (see CONFIG.md). Non-dev creators rely on SETUP.md's guided edits.

## ADR-004: Page in React; widget (v1) as a framework-free Web Component
**Context:** Owner is fastest in React. But an embeddable widget must drop into WordPress, Hugo, plain HTML, Vue sites — shipping React runtime to every host page is unacceptable (~45KB+ and version conflicts).
**Decision:** v0 page: Vite + React. v1 widget: Lit (or vanilla) Web Component sharing a framework-free `core` (upi.ts, schema) via pnpm workspaces. `src/lib` and `src/config` are written React-free from day one to make extraction mechanical.
**Consequences:** Two UI implementations of the payment card eventually. Accepted: the card is small, and the alternative (React everywhere) kills the embed story.

## ADR-005: GitHub Pages as the default deploy target; Vercel as the documented alternative
**Context:** Both are free. Pages needs zero new accounts for the target user, reinforces "you own everything", and survives if any single company changes free-tier policy. Vercel has nicer DX and previews.
**Decision:** Ship `deploy-pages.yml` as the batteries-included path (subpath-aware via `github.event.repository.name`); README carries a Deploy-to-Vercel button.
**Consequences:** All asset paths must respect Vite `base` (CI enforces by building with a subpath). Two documented paths to keep green in CI.

## ADR-006: Deeplink is best-effort; QR + Copy-VPA are the guaranteed paths
**Context:** GPay/PhonePe deliberately degrade browser `upi://` intents to unverified VPAs (silent no-ops, "limit exceeded" errors, ignored amounts). Failure is undetectable directly.
**Decision:** Mobile shows all three methods; deeplink gets a visibility-change heuristic (1.5s) that, on suspected failure, surfaces a friendly nudge toward Copy-VPA/QR. Marketing copy never promises one-tap payments.
**Consequences:** Slightly busier mobile pay zone; materially higher real-world success rate; honesty as brand.

## ADR-007: Analytics behind an adapter, off by default, intent-only events
**Context:** Analytics is genuinely useful (creators asked: views, amount impressions) but is also the main privacy risk and the only network egress. Completions are untrackable (ADR-001).
**Decision:** Three-event contract (`page_view`, `amount_selected`, `pay_clicked{method}`) behind an `AnalyticsAdapter`. Default noop with zero network and zero bundled bytes (lazy `import()` of PostHog only when configured). Donor message content is never an event property. PostHog first (free 1M events/mo, custom events); Umami/Plausible as community adapters.
**Consequences:** "Amount impressions ≠ payments" must be explicit in creator docs. Adapter interface invites contributions without expanding our surface.

## ADR-008: Mandatory documented ₹1 self-test before going live
**Context:** A VPA typo that still matches the format regex sends every donation to a stranger, unrecoverably. No API exists to verify VPA ownership without a PSP.
**Decision:** SETUP.md's launch checklist ends with a required step: scan your own deployed QR and send yourself ₹1 from a different UPI account/app. The README badge/checklist references it.
**Consequences:** Can't be technically enforced; treated as a documentation/culture guarantee. Build-time regex + `.strict()` catch the mechanical errors.

## ADR-009: Single-package repo for v0; workspaces only when the widget lands
**Context:** Temptation to scaffold the "correct" monorepo now.
**Decision:** v0 is one package. Restructure to pnpm workspaces in the first v1 PR.
**Consequences:** One `git mv`-heavy PR later; much lower fork/setup friction for the entire v0 life, which is when most forks happen.

## ADR-010: UPI URI encoding is RFC 3986 — never `URLSearchParams`
**Context:** ARCHITECTURE.md originally specified `URLSearchParams` encoding for the intent URI. `URLSearchParams` implements the WHATWG `x-www-form-urlencoded` serializer, which encodes a space as `+`. UPI apps decode `upi://` query strings with plain percent-decoders, where `+` is a literal plus (RFC 3986 §3.4). Verified: `new URLSearchParams({tn:'Thanks for the chai'}).toString()` → `tn=Thanks+for+the+chai`.
**Decision:** Encode with `encodeURIComponent`, additionally escaping `!'()*` so the payload is unreserved-only. Apply encoding to `pn` and `tn` **only**. Emit `pa`, `am` and `cu` verbatim — every character a regex-valid VPA can contain is RFC 3986 unreserved or `@`, and percent-encoding the `@` (`shivam%40okaxis`) would break parsers that don't decode `pa`.
**Consequences:** Donor notes and creator names render correctly in the payer's app instead of showing `+` for every space. `am` never gets a `%2E`. Supersedes ARCHITECTURE.md's earlier `URLSearchParams` wording, which is now corrected. A regression test asserts `encodeUpiComponent('a b') === 'a%20b'` **and** that `URLSearchParams` still differs, so the trap is locked shut.

## ADR-011: Integer rupees only, plus a numeric-integrity hard ceiling
**Context:** ARCHITECTURE.md left "1.5 → reject?" open. Meanwhile `(1e21).toFixed(2)` returns `"1e+21"`, which would emit a malformed `am`.
**Decision:** Reject fractional rupees (`AMOUNT_NOT_INTEGER`). Add `HARD_MAX_AMOUNT_RUPEES = 10_000_000` (₹1 crore) as a numeric-integrity guard with its own code, distinct from the ₹1,00,000 soft cap. The soft cap stays a UI warning surfaced via an `exceedsSoftCap` flag on the success value — the builder never errors on a large amount.
**Consequences:** Aligns with DESIGN.md, which already says "integers only (UPI supports paise but donors think in rupees)". Every amount the product can produce is an integer by construction (`basePrice` and `presets` are both integers). Avoids the silent class of failure where an app rounds or drops paise and we cannot detect it (ADR-001: no confirmation channel). `chai.maxAmountWarning` remains creator-configurable.

## ADR-012: The `tn` limit is 60 decoded code points, not an encoded-byte budget
**Context:** CLAUDE.md says "`tn` ≤ 60 chars"; ARCHITECTURE.md said "truncated to 60 chars post-encoding budget". These conflict. A post-encoding budget would allow a Devanagari note roughly 6 characters (each is 9 encoded bytes: `%E0%A4%9A`) and an emoji note about 5.
**Decision:** 60 **decoded code points**. Truncate via `Array.from`, never by UTF-16 slicing.
**Consequences:** Usable for the Hinglish/Indic audience this product targets, and it matches the character counter the donor reads in the UI. UTF-16 slicing is banned because splitting a surrogate pair yields a lone surrogate and `encodeURIComponent` throws `URIError` on one — that would crash the QR render path on a keystroke. Sanitising strips invisible and bidi-spoofing characters but deliberately **preserves** U+200C ZWNJ and U+200D ZWJ, which are semantically required in Devanagari conjuncts and family emoji.

## ADR-013: The placeholder guard is a deploy-time gate, not a build-time one
**Context:** The example config must ship with a creator-shaped VPA. `yourname@bank` is a valid *format*, so the Zod schema accepts it — which is required, because CI has to build the shipped example to prove the example is valid. But a fork that deployed it unedited would publish a page whose QR points at nobody.
**Decision:** Keep `pnpm build` pure so the canonical repo and every contributor fork build the example green. Add `pnpm build:deploy` = `check:placeholder && vite build`, used by the Pages workflow. The canonical repo's CI asserts the guard **rejects** the shipped example (a negative test) — that is what proves the guard is wired up. `CHAI_ALLOW_PLACEHOLDER=1` is the documented escape hatch for previewing a deploy.
**Consequences:** Two build entry points to keep straight. Detection is case- and whitespace-insensitive, so "editing" only the capitalisation still fails. The canonical repo must not publish a live demo from the placeholder config; if a public demo is ever wanted it needs either a real VPA or an unmissable "example only — do not send money" banner, per ADR-008's spirit.

## ADR-014: Toolchain — exact pins, current majors, Node 24
**Context:** ARCHITECTURE.md specified Vite 6 and CLAUDE.md specified Node 20+. At scaffolding time the registry had moved considerably, and `@vitejs/plugin-react@6` declares `peerDependencies: { vite: "^8.0.0" }` — Vite 6 and 7 will not install with it.
**Decision:** Vite 8, TypeScript 7, Zod 4, Vitest 4, Tailwind 4, Node 24 (`engines`, `.nvmrc`, `.node-version`, CI `node-version-file`). React stays pinned to 18.x including `@types/react` 18.x, per ADR-004. All dependencies are exact-pinned, no carets.
**Consequences:** Records a visible deviation from a written doc, forced by a hard peer constraint rather than preference. `@types/react` must never drift to 19.x while React is 18. `@vitest/coverage-v8` must track `vitest` exactly. Node 24 strips TypeScript types natively, which is why `scripts/check-config.mts` needs no `tsx`/`ts-node` dependency.

## ADR-015: The framework-free core emits error codes and is exempt from the `strings.ts` rule
**Context:** CLAUDE.md requires all user-visible strings to live in `src/strings.ts`. But `src/lib/upi.ts` and `src/config/*` must stay import-free so they can be extracted to `packages/core` for the v1 widget (ADR-004), and config errors are creator-facing build output rather than page copy.
**Decision:** The core emits a stable `UpiErrorCode` plus developer-facing English; UI copy is keyed off the code in `strings.ts`. Config validation messages live in the schema. The `Buy {name} a chai` `meta.title` default is derived in the schema too, because it is needed before React mounts.
**Consequences:** Two documented exceptions to a hard convention, noted in the `strings.ts` header so nobody "fixes" them. i18n later translates by code, not by string matching. Note the deliberate layering difference: the schema *rejects* a `creator.name` over 50 characters at build time, while `sanitizeName` *truncates* at runtime — build-time and runtime have different jobs.

## ADR-016: `defineConfig` is an identity function; two error surfaces by design
**Context:** `defineConfig` could validate at authoring time. If it did, an invalid config would throw a raw `ZodError` during module evaluation, before `load.ts` could format it.
**Decision:** `defineConfig` is a typed identity function and never parses. Validation happens exactly once, in `parseConfig`. Build enforcement is the `chai-config-validator` Vite plugin, because a bundler only bundles modules — it never executes them, so a module-scope `throw` in app code does **not** fail `vite build`.
**Consequences:** Creators always see the CONFIG.md-formatted block, never a Zod stack. TypeScript catches shape and typo mistakes in the editor; Zod catches value mistakes at build; neither can do the other's job. `src/config/load.ts` is kept side-effect-free so build scripts can `try`/`catch` it, with the app's singleton isolated in `src/config/config.ts`.

## ADR-017: We render the QR ourselves; `qrcode` is used only as an encoder
**Context:** ARCHITECTURE.md specified `qrcode` "(client-side canvas → dataURL)". Its `toCanvas`/`toDataURL` renderers require a real `<canvas>`, which jsdom does not implement. That makes the download path — an explicit P0.5 acceptance criterion ("downloadable as PNG") — untestable in CI, on a payment surface where a wrong QR sends money to a stranger unrecoverably (PRD §8.3).
**Decision:** Use only `qrcode`'s pure-JS `create()` to get the module matrix, and render both outputs ourselves in `src/lib/qr.ts`: an SVG for display and a hand-rolled 1-bit greyscale PNG for download. No canvas anywhere.
**Consequences:** `qr.test.ts` decodes the exact bytes a donor downloads, with `jsQR` reading the pixels back and Node's `zlib.inflateSync` independently validating our zlib stream — neither is our own code marking its own homework. Display and download derive from one matrix, so they cannot disagree. The cost is ~90 lines of encoder and a PNG that is roughly 10× larger than a compressed one (~14KB, using stored DEFLATE blocks rather than implementing Huffman coding) — paid once, on a button a donor presses at most once. `qr.ts` stays framework-free and DOM-free, so it extracts to `packages/core` with the rest (ADR-004). Supersedes ARCHITECTURE.md's "canvas → dataURL" wording, now corrected.

## ADR-018: The brand accent is a fill colour, not a text-on-fill colour
**Context:** The payment card's selected preset chip was originally a solid `--chai-accent` (#C4622D) fill with white text. Measured with the project's own `contrastRatio`, white on that accent is **4.09:1** — it clears WCAG 1.4.11 (3:1) for a UI component fill, but fails 1.4.3 (4.5:1) for the chip's label-sized text sitting on top of it. This is the same 4.09:1 that ADR-014-era work already established is within 0.5% of the theoretical maximum for this hue against both light and dark surfaces, so "pick a better terracotta" is not available.
**Decision:** Selected state is `--chai-accent-soft` tint plus a 2px `--chai-accent` border with ink text (13.9:1). A separate `--chai-accent-strong` (#A34E22, 5.7:1 on white) is defined for any surface that genuinely needs white text on a filled accent — the mobile "Pay with UPI app" button in Session 3 is the expected first user. The accent itself is reserved for borders, icons, focus rings and **large** text such as the amount numeral, where 3:1 is the applicable bar.
**Consequences:** DESIGN.md's `--chai-accent-ink: #FFFFFF` pairing is valid only against `accent-strong`, never against `accent`. Any future component putting normal-size text on the accent is an accessibility regression; the tokens are named so the correct one is the obvious one to reach for. Dark mode lifts the accent to #E08A4F (6.4:1 on the dark surface), where the constraint does not bind.

## ADR-019: The pay-zone layout turns on one device heuristic; the deeplink follows it
**Context:** ARCHITECTURE.md listed `isMobile()` and `canDeeplink()` as two device heuristics. In v0 they answer the same question. A `upi://` intent only resolves where a UPI app is installed — Android and iOS — which is exactly the coarse-pointer, phone-or-tablet population where a QR is useless because a device cannot scan its own screen. There is no device that is "mobile but cannot deeplink" or "desktop but can," so a second predicate would be an alias.
**Decision:** One heuristic, `isMobileDevice()` in `src/lib/device.ts`, resolved in confidence order: a positive `navigator.userAgentData.mobile`, then `matchMedia('(pointer: coarse)')` (which also catches tablets the client hint reports as non-mobile), then a UA sniff. `useIsMobile` binds it through `useSyncExternalStore` so the layout follows a live pointer change — a mouse plugged into a 2-in-1, or devtools emulation. `PayZone` leads with buttons + deeplink when it is true and with the QR when false; the deeplink is shown iff mobile.
**Consequences:** The three paths stay peers regardless of branch (hard rule 3) — copy and QR never hinge on the heuristic. A false "desktop" on a touch laptop just shows the QR, which still works; a false "mobile" shows a deeplink that may no-op, which the visibility heuristic (ADR-006) already catches and softens. `device.ts` is DOM-touching, so it sits beside `download.ts` in `src/lib` rather than in the framework-free core. Supersedes ARCHITECTURE.md's two-function wording, now corrected.

## ADR-020: The `<noscript>` block carries the real VPA, injected at build time
**Context:** Copy-UPI-ID is the one guaranteed payment path (ADR-006), and with JavaScript disabled it is the *only* one: the QR is built client-side and React never mounts. A generic "copy the UPI ID" sentence is useless without the actual ID, and the ID is not in the served HTML — it lives in `chai.config.ts`, which the static `index.html` cannot see.
**Decision:** A `chai-noscript` Vite plugin (`transformIndexHtml`) parses the config at build and replaces the `<noscript>` in `index.html` with the creator's real VPA and derived title, inline-styled so it renders without the bundle. Name, title and VPA are HTML-escaped — the VPA regex already forbids HTML-special characters, but a display name does not. Any failure leaves the static fallback noscript from `index.html` in place. The copy lives in the plugin, not `src/strings.ts`, for the same reason `meta.title`'s default lives in the schema (ADR-015): it is needed before — or entirely without — a React mount, so it cannot depend on the UI string layer.
**Consequences:** A JS-disabled donor sees the exact UPI ID to type into their app, on-thesis with "VPA typos are catastrophic; copy-VPA is the guaranteed path." A build-time *static QR* remains a logged nice-to-have (DESIGN.md §Empty/edge). The plugin runs in both serve and build but is guarded so a config error never blocks dev — the config validator plugin and `src/config/config.ts` already own that failure surface.

## ADR-021: The config accent is derived into contrast-safe tokens; `theme.mode` pins the palette via `data-theme`
**Context:** DESIGN.md promised `theme.accent` overrides and a `theme.mode` of `light | dark | auto`, but `index.css` only carried one hand-tuned accent per surface. A single config accent cannot simply be assigned to `--chai-accent`: `--chai-accent-strong` is the *only* token allowed to carry white text (ADR-018), and a light accent would fail AA there, while dark mode needs the accent *lifted* to stay legible on the dark surface. And forcing a mode cannot be a bare inline `style="--chai-*"` on `:root` — inline styles beat the `prefers-color-scheme` media rule regardless of the OS, which would break the auto default.
**Decision:** `src/lib/theme.ts` derives four tokens per surface from the parsed accent: the accent as-is; `-strong` darkened until white clears 4.5:1 (light) or the accent lifted until it clears 4.5:1 on the dark surface (dark, where the ink is the dark surface); `-soft` a heavy tint; `-ink` white or the dark surface. The override is one `<style>` that mirrors index.css's *exact* selectors — `:root`, `@media (prefers-color-scheme:dark) :root:not([data-theme="light"])`, and `:root[data-theme="dark"]` — appended after index.css so equal specificity resolves in our favour by source order, including under a forced dark theme. The built-in terracotta short-circuits to `null` (index.css already carries the hand-tuned tokens, which beat any derivation). A modern colour function that cannot be parsed to RGB (`oklch()`, `lab()`) falls back to `color-mix()` best-effort — the build already warns its contrast is unverified. `theme.mode: 'light' | 'dark'` stamps `data-theme` on `<html>`: at build by `chai-head` (flash-free) and re-affirmed at runtime by `applyTheme`; `'auto'` removes it and leaves the media query in charge.
**Consequences:** Any accent yields a legible white-text "Pay" button and readable chips in both themes, with no per-mode values asked of the creator. The two dark token blocks in index.css (the `@media` one and the `[data-theme="dark"]` one) are identical and must stay in sync. Runtime accent injection is a one-frame flash for non-default accents only; forced mode is flash-free. `theme.ts` is DOM-touching, so it sits in `src/lib` beside `device.ts`, out of the framework-free core; the derivation itself is pure and unit-tested.

## ADR-022: The document head — title, description, OG/Twitter, `lang`, forced theme — is baked in at build time
**Context:** Open Graph / Twitter cards and the `<title>` must live in the *served* HTML: social crawlers and search bots do not execute the bundle, so a head set from React is invisible to exactly the consumers that need it. `main.tsx` previously set `document.title` and `<html lang>` at runtime, which browsers honour but crawlers never see.
**Decision:** A `chai-head` Vite plugin (`transformIndexHtml`) parses the same config and *replaces* `<title>`, the description `<meta>` and the `<html lang>` attribute, then injects `og:*` / `twitter:*` / `theme-color` tags and stamps `data-theme` for a forced `theme.mode`. String replacement rather than the `tags` API because a title/description must be replaced, not appended (a second `<title>` is ignored). `og:image` resolves a root-relative path against `BASE_PATH`; a creator who wants reliable cards sets an absolute `https://` URL, the only form crawlers fetch cross-host (CONFIG.md notes this). Any failure leaves index.html's static head untouched — the same contract as `chai-noscript`. `main.tsx` no longer touches title/lang.
**Consequences:** Shared links render the right card and title with no JS. The accent stays a runtime concern (visual-only, crawler-irrelevant). Two build-time config parses now run (`chai-noscript` + `chai-head`); the cost is negligible. The description fallback lives in the plugin, not `src/strings.ts`, for the same reason as ADR-015/ADR-020 — it is needed before, or entirely without, a React mount.

## ADR-023: Brand logos come from `simple-icons`; Lucide covers only UI glyphs
**Context:** DESIGN.md §Iconography specified a single Lucide icon set. Lucide 1.x dropped every brand mark — `Github`, `Twitter`, `Youtube`, `Linkedin` and the rest are gone — so the socials row (P0.2) had no logos to render.
**Decision:** Brand marks come from `simple-icons` (CC0, path-data only: no CDN, nothing over the network, so the privacy rule holds). Only the ~30 mapped brands are named-imported, and the package is side-effect-free ESM, so the bundle carries those path strings rather than the full 3,000-icon set. `src/lib/social.ts` maps a URL's registrable domain to a brand mark or to a generic kind; the component draws the brand path, or a Lucide `Rss`/`Globe` for feeds and unknowns. A brand `simple-icons` no longer ships (LinkedIn) degrades to the globe, still carrying its config `label`. Lucide remains the UI-glyph set (chevron, copy, download, smartphone, etc.).
**Consequences:** Recognisable logos without shipping possibly-inaccurate hand-authored SVG paths (a mangled glyph is worse than none). The `label` always carries identity, so a mis-detected or unmapped domain is never blank. `simple-icons` is a pinned dependency. Amends DESIGN.md §Iconography's "single set (Lucide)".

## ADR-024: The desktop page is two columns — identity + projects left, a pinned payment ticket right; mobile is one column
**Context:** DESIGN.md §anatomy specified one centred 480px column at every width — identity → trust → payment → footer, stacked. On a desktop that wastes the width and pushes the QR below the fold, exactly where the QR is the primary path (a second phone scans it, ADR-019). The owner asked for a desktop layout with profile + projects on the left and the QR always visible on the right, plus an explicit mobile order: masthead → profile → payment → projects → footer.
**Decision:** One CSS grid, `.chai-grid`, driven by `grid-template-areas`. Mobile is a single column in the exact DOM/reading/tab order profile → payment → projects. Desktop (≥1024px) is two columns: profile and projects fill the left column while the payment ticket takes its own 380px right column, spanning both left rows, with its inner wrapper `position: sticky` so the QR stays visible as the left column scrolls. Template areas rather than source reordering keep the DOM in reading order (which is also the mobile order and the tab order). The masthead and footer sit *outside* the grid so they stay full-width; masthead, grid and footer share one `max-w-[480px] lg:max-w-[1040px]` shell so their left edges align. The payment card keeps its own ≤480px max-width, so it is still a clean lift-out for the v1 widget.
**Consequences:** Supersedes DESIGN.md §anatomy's single-480px-column and "Max content width 480px" lines. The pay-zone device branch (ADR-019) is unchanged — desktop still leads with the QR, which is precisely what "always visible on the right" wants, and mobile still leads with buttons with the QR one tap away. Sticky assumes the left column is the taller one, which it normally is (identity + bio + projects); when it is not, the card simply doesn't scroll, which is fine. Below 1024px — tablets included — the layout is the mobile single column, chosen over a cramped two-column at 768px where a 380px card starves the left.

## ADR-025: The core palette is brand-locked; only the accent is themeable; the canvas is off-white
**Context:** ADR-021 lets `theme.accent` recolour the accent tokens, but DESIGN.md's canvas was a warm milk-cream (#FFF8F0) and a Session-4 pass had added an accent-tinted radial glow to the `body` and a sheen to the card. The owner wants the project's core identity to stay the same for every fork — for branding — with only the CTA, borders and accent text customisable, and asked for a very whitish / off-white canvas in light mode with a suitable dark counterpart.
**Decision:** The core palette — `--chai-bg`, `--chai-surface`, `--chai-ink`, and their muted/line companions — is brand-locked and is *not* exposed in the config. `theme.accent` stays the single visual customisation and only ever touches the four accent tokens (fills/CTA, borders, focus rings, accent text), derived exactly as ADR-021 describes. The light canvas moves from cream to a warm off-white `--chai-bg: #faf8f4`; the surface stays #FFFFFF. Dark mode keeps the tuned warm charcoal (#1A130E / #241B14), which is already suitable. The body glow and the card sheen are removed: a tinted canvas is the one thing an arbitrary custom accent would fight, so the background stays flat. Warmth now lives in the accent, the dark-brew ink and the ☕ marks. Only the light `--chai-bg` changed value — the accent-derivation surfaces (#FFFFFF and #241B14 in `theme.ts` / `warnings.ts`) and every measured contrast pairing are untouched.
**Consequences:** Any accent stays legible against a fixed, predictable canvas, and a creator cannot ship an illegible page by tinting the background — there is no knob to tint it. Supersedes DESIGN.md §Visual language's cream `--chai-bg` and its "ambient warmth" bullet. The page's signature shifts from atmosphere to structure — the pinned live payment ticket of ADR-024 and the perforated tear. `theme-color` (light) updates to #faf8f4.

## ADR-026: Project branding is a locked masthead wordmark plus two deletable footer links
**Context:** DESIGN.md §4 defined project branding as a single tiny footer line linking to the repo. The owner wants the project's identity carried in a masthead, and wants two template links present by default — the project's GitHub page and the maintainer's own Buy Me a Coffee page — while accepting that any fork can delete them from the public source.
**Decision:** A locked `Masthead` banner carries the project wordmark (`strings.brandName`, "buy me a chai"), identical on every fork, framing the creator's page as "made with buy me a chai". It is the page's banner landmark and deliberately *not* an `<h1>` — the creator's name is the single heading — so it is a logotype in plain text. The old `Header` component, which was always the creator identity block and never the page header, is renamed `Profile` and drops its banner role; `Masthead` is the real page header now. The footer carries two links, both in `strings.ts` (ordinary UI copy, not config) and both deletable from source: `poweredBy` → the repo, and `supportMaker` → the maintainer's Buy Me a Coffee page. They ship on as the only ask of a free project. The maker URL is a template default (`buymeacoffee.com/shivams136`) a fork replaces or removes, exactly like the placeholder VPA.
**Consequences:** Supersedes DESIGN.md §4's "single small footer line" and §anatomy's footer — two links now, still tiny, still never configurable-off in spirit (a fork edits source, it has no toggle). One banner landmark (the masthead) and one contentinfo (the footer); the profile is a plain block carrying the h1. Branding copy lives in `strings.ts` beside the rest of the UI strings, so a future i18n pass translates it by the same mechanism.

## ADR-027: Project & maker identity live in `src/project.ts`; branding links carry referral attribution; PostHog capture deferred to Session 5
**Context:** ADR-026 put the repo and support URLs as literal strings in `strings.ts`. The owner wants a single, find-and-replace-friendly home for the project author's identity, the branding links surfaced in the masthead as well as the footer, and referral attribution — the project is a GitHub *template*, so every clone is a billboard, and the maker wants clone-driven traffic to be traceable without a backend. They also want an inbound source shown in the UI, and UTM captured in PostHog. Two directions were chosen (both), and the PostHog capture deferred.
**Decision:** A dedicated `src/project.ts` holds `MAKER` (`name` for display, `handle` for compact/URL use — the two name variables) and `MAKER_PROJECT` (`name`, `repoUrl`). The names are deliberate: `MAKER`/`MAKER_PROJECT` are the template author and their repo, as opposed to the `CREATOR` and their `CREATOR_PROJECT`s (`creator` / `works` in `chai.config.ts`) — a reader never has to guess which actor a value belongs to, and the README documents the four. This is the origin project's branding, not per-creator config, so it stays out of `chai.config.ts` — a fork edits these constants or deletes the links. The masthead carries one labelled CTA — "Create your support page" — pointing at `MAKER_PROJECT.templateUrl` (GitHub's use-this-template flow) rather than the bare repo: the masthead is the page's growth loop, so it sells the template, not the source. The footer carries both the repo credit (`MAKER_PROJECT.repoUrl`) and the support link (`MAKER.supportUrl`). `src/lib/referral.ts` provides two pure, injectable-argument helpers: `withReferral(url, surface)` appends `utm_campaign` = the **source project** (so every clone groups together), `utm_source` = this deployment's host (which clone), `utm_medium=referral`, `utm_content` = surface, and `ref` = host — the host is public, never PII; and `readInboundSource(search)` reads `ref`/`source`/`utm_source`, sanitises (drop control + angle chars, cap 48) and returns plain text for a quiet `ReferralNote` chip. **Capturing UTM/source in PostHog is deferred to Session 5** (deploy + analytics): analytics is not yet wired into any component, and adding these as event properties changes the fixed 3-event contract (ANALYTICS.md) — that lands with the rest of analytics, keyed off the same `readInboundSource` signal.
**Consequences:** One edit point for a fork's identity; the maker name appears at primary (display) and secondary (handle) places via two variables. Outbound links carry attribution with no network call — href construction only, so hard rule 4 holds and PostHog stays untouched until Session 5. The inbound chip renders sanitised text, never a URL or markup, so a crafted `?ref=` cannot inject. Amends ADR-026: the URLs move from `strings.ts` to `src/project.ts`, and the masthead gains a "Create your support page" CTA into the template flow (the support link stays footer-only, so the masthead is a clean brand bar). Tagging that CTA is what makes clone intent measurable at the exact moment it happens. `strings.poweredBy`/`supportMaker` become functions of the project/maker name so the constants flow through the copy layer without `strings.ts` importing `project.ts`.

## ADR-028: A disabled build contains no analytics code, not merely no analytics calls
**Context:** Hard rule 4 requires zero network calls when analytics is off. A runtime `if` satisfies that, but `posthog-js` still lands in `dist` — ~220 kB (73 kB gzipped), roughly doubling the download of a page whose whole pitch is that it is small, static and auditable. Analytics is off by default, so *most* forks would be paying that. `chai.config.ts` is known at build time, which makes the stronger guarantee available for the cost of one `define`. A weaker option — leaving the decision at runtime and documenting the chunk as harmless — was rejected: "no network calls" is a claim a reader has to take on trust, while "no bytes" is a claim they can check with `ls`.
**Decision:** A `chai-analytics-flag` Vite plugin injects `__CHAI_ANALYTICS__`, true only when the *raw* config declares an `analytics` block (`declaresAnalytics` in `load.ts` — the parsed config always reports disabled under plain Node, where no `import.meta.env` exists). `src/analytics/index.ts` gates `import('./posthog.ts')` behind that flag **lexically**, in a ternary; passing the flag as an argument alongside a loader closure reads better and does not work, because the closure is constructed unconditionally and Rollup then cannot prove the import is unreachable. Two gates, not one: the flag decides which bytes ship, the runtime `config.analytics` check decides whether they run. The SDK itself is initialised with autocapture, session recording, heatmaps, dead clicks, rageclick, exception capture, web vitals, surveys, web experiments, feature flags and external-script loading all explicitly off, plus a `before_send` that drops any event whose name is not one of the three in ANALYTICS.md.
**Consequences:** A default fork's `dist/` contains the string "posthog" only inside Zod error messages. The mechanism is invisible and easy to regress, so it is guarded twice: a CI step greps `dist/` for SDK markers, and `src/analytics/contract.test.ts` asserts the import stays inside the branch. `posthog-js` is a plain dependency (installed by every fork, shipped by none that has analytics off) and drags in `core-js`, whose funding postinstall is denied via `pnpm.ignoredBuiltDependencies` so installs stay non-interactive. Enabling analytics costs one extra 220 kB chunk, loaded lazily and never on the payment path. Session recording being off is not a tuning choice — it is the line that keeps a donor's typed message out of a replay, and it is why the config is explicit rather than default.

## ADR-029: The Pages deploy derives its base path from the repository name, with one variable for custom domains
**Context:** Hard rule 7 says every build must work on a Pages subpath. Hardcoding `/buy-me-a-chai/` breaks the moment a creator renames their fork — the most common first edit — and the failure mode is a blank page with 404s on every asset, which is the single hardest thing for a non-developer to diagnose. Custom domains have the opposite need: they serve from the root, so any subpath prefix breaks them identically.
**Decision:** `deploy-pages.yml` sets `BASE_PATH` to `${{ vars.CHAI_BASE_PATH || format('/{0}/', github.event.repository.name) }}`. Renamed forks work untouched; a custom domain sets one repository variable to `/`. `build:deploy` (not `build`) runs, so the placeholder guard of ADR-013 gates every publish. Deploy does not re-run lint and tests — CI already runs on the same push, and a creator editing `chai.config.ts` should not wait for a test suite to see their page. The optional PostHog key is read from `vars` and falls back to `secrets`, because a value that looks like a key gets put in Secrets no matter what the docs say.
**Consequences:** One workflow serves every fork with no edit. The `CHAI_BASE_PATH` escape hatch is the only deploy knob, and it is documented in SETUP.md next to the CNAME step. CI keeps building the subpath layout and asserting the rewrite, so a regression is caught on the PR rather than on someone's live page.

## ADR-030: Creator values move to `chai.config.yaml`; validation moves to build time
**Supersedes the file-format half of ADR-003 (the Zod schema and its guarantees are unchanged).**
**Context:** ADR-003 chose a typed TS config for autocomplete, comments and loud typos, and those arguments still hold — but it assumed the creator and the template edit the same file, which is what makes every template update a merge conflict in the one file a creator cannot afford to lose. Keeping `chai.config.ts` conflict-free would require promising never to edit the shipped example again; a promise enforced by a CI guard is strictly weaker than a structure where the two sides simply do not share a file. `.env` was considered and rejected: `works` and `socials` are arrays of objects, and flattening them to `CHAI_WORK_1_TITLE` would make the creator's experience worse than the thing it replaces. This is a breaking change to the public API (CONFIG.md), which is why it happens before v0.1.0 rather than after.
**Decision:** Creator values live in `chai.config.yaml` — comments and nesting both survive, and a `chai.schema.json` generated from the Zod schema by `z.toJSONSchema` (native in Zod 4) gives autocomplete and inline field docs in VS Code *and* github.dev, which is better than `defineConfig` for a creator who never opens a terminal. A `chai-config` Vite plugin reads the YAML, validates it with Zod at **build time**, and exposes the result as `virtual:chai-config`. The browser therefore receives a plain object: Zod and the YAML parser both leave the bundle. The analytics key stops being `import.meta.env?.VITE_POSTHOG_KEY` written by the creator and is injected by the plugin from the environment — removing the `?.` footgun that broke `pnpm typecheck` the moment anyone followed SETUP.md's own analytics instructions.
**Consequences:** A creator's repo diverges from the template in exactly one text file plus `public/`, so `git merge upstream/main` is conflict-free by construction rather than by discipline. TypeScript no longer catches config shape errors — Zod and the JSON Schema carry that alone, so the schema needs `.describe()` on every field to keep the editor useful, and CI must assert the committed `chai.schema.json` still matches the Zod schema or the autocomplete quietly starts lying. `src/config/*` stays framework-free and extractable for the v1 widget; the file reading lives in `scripts/read-config.mts`, outside the core, because it needs `node:fs`.

## ADR-031: Updates are a button, not a terminal; editing source opts you out
**Context:** A template repo shares no history with the repos made from it, so a creator's first `git merge upstream/main` needs `--allow-unrelated-histories` — and more to the point, needs a terminal, a clone and Node, none of which SETUP.md asks for. Onboarding is fully browser-based and updating must be too, or the second half of a creator's life with this project is strictly harder than the first.
**Decision:** `update-template.yml` runs on `workflow_dispatch` — a **Run workflow** button. It merges upstream onto a branch, unconditionally restores the creator's `chai.config.yaml` and `public/`, runs build and tests in the same job, and opens a PR. Two clicks, no terminal. Files under `.github/workflows/` are **excluded from the PR** and listed in its body instead: `GITHUB_TOKEN` is forbidden from writing workflow files, and the alternative — asking every creator for a PAT with `workflow` scope — trades a rare manual copy for permanent onboarding ceremony and a token that expires into silent failure. No `schedule:` trigger: scheduled workflows are auto-disabled after 60 days of repository inactivity, and a donation page is inactive by design, so a cron would look like it works and then quietly stop. **A creator who edits source files is a developer and out of scope for this path** — the merge prefers upstream for code, the PR says so plainly, and anyone stripping branding from `Masthead.tsx`/`Footer.tsx` owns their own merges.
**Consequences:** Updating costs two clicks for the creators the project is written for, and nothing at all for the majority who never update — a static page that works does not rot. The support boundary is documented rather than discovered. The workflow-file gap is a real hole: releases that change CI need a manual step, and if that turns out to be most releases, the PAT option is the escape hatch to revisit.

## ADR-032: Branding values are config with maker defaults; the links stay in source
**Amends ADR-026 and ADR-027.**
**Context:** ADR-027 put `MAKER`/`MAKER_PROJECT` in `src/project.ts`, reasoning that origin branding is not per-creator config. True — but it leaves a source file a rebranding creator must edit, which under ADR-030's one-file rule is the only remaining structural conflict.
**Decision:** The values move into the schema as a `branding` block whose **defaults are the maker's own**, overridable in `chai.config.yaml`; `src/project.ts` goes away. `Masthead.tsx` and `Footer.tsx` still hold the links themselves. Deleting them is still a code edit, deliberately: ADR-026's point was that removal should be an act, not a flag someone flips while skimming a config file, and that survives intact.
**Consequences:** Rebranding is a config edit; removal is still a source edit and still opts you out of ADR-031's update path. Because the defaults live in template-owned source, the maker changing their own support URL propagates to every fork on update, which is the behaviour we want and the opposite of what a copied value would do.
